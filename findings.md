# 项目调研与最新发现

## OCR 产品方向结论

### 1. 三种识别模式应当承担不同职责

- `tesseract`
  - 适合纯文本、轻量、低成本、快速识别
  - 适合“先拍先录”的基础场景
- `pix2text_online`
  - 更适合公式类题目
  - 适合作为公式识别增强方案，而不是整页复杂题主方案
- `doubao_vision`
  - 更适合复杂题、混排题、整页结构化理解
  - 更适合作为“高复杂度题目”的主识别方案

### 2. 题目中的附图不应强行转成文字

对于流程图、装置图、结构图这类内容：

- 正文识别负责把题干和选项识别清楚
- 图本身应尽量作为图片保留
- 实在无法自动裁出时，应允许老师在编辑入库页补拍

这比把附图解释成一段英文说明更符合老师真实使用习惯。

## 本轮实现后的关键发现

### 1. 结果页正文不完整的根因不在模型本身

这次问题的根因，主要不是豆包整体识别能力差，而是前端结果页取正文的逻辑不稳定：

- 之前会在多个字段里“挑最长的一段”来展示
- 这会导致结构化题干、blocks、纯文本结果之间出现误选
- 结果页和编辑页又没有共用同一套规则，所以会看起来“不一致”

### 2. 正确做法是统一正文规则

现在已经改成：

- 结构化题干优先
- 自动补上结构化选项
- 必要时再回退到 blocks / result_text / result_latex

这样结果页和编辑页都用同一套正文规则，后续定位问题会清楚很多。

### 3. “置信度”更适合做弱提示，不适合当核心判断

当前更合理的产品表达是：

- `识别质量：较高`
- `识别质量：一般`
- `识别质量：需复核`

并附带 `仅供参考`。

对于老师来说，“要不要复核”比“0.87 还是 0.91”更有用。

### 4. “建议补拍附图”应当是规则判断，不是原样展示模型说明

当前更合理的判断逻辑是：

- 模型判断题目有图，且建议保留原图
- 或正文中出现“如图 / 流程图 / 装置图 / 结构图”等关键词
- 且当前题目还没有可用附图

满足这些条件时，前端给老师一个明确动作提示：

- `建议补拍附图`
- `去补拍附图`

### 5. OCR 附图真正入库，需要后端一起配合

仅仅在前端显示图片还不够。

如果老师从 OCR 结果进入编辑入库：

- OCR 已识别出的附图需要真正挂到题目上
- 老师手动补拍的附图也要一并保存
- 详情页和后续导出能力才能复用这些图片

本轮已经把这条挂接链路补上了一版。

## 当前产品判断

### 结果页仍然有保留价值

结果页的定位应当是：

- 看原图
- 看识别结果
- 改几个错别字 / 漏字
- 没大问题就快速保存
- 复杂问题再进编辑入库

所以结果页不是“多余页面”，但它不应该再承担过多技术说明或重复编辑负担。

### 编辑入库页承担完整维护职责

编辑入库页更适合承担：

- 完整编辑题目
- 维护答案、解析、题型
- 维护题目附图
- 补拍附图
