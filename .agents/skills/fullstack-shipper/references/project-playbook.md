# 项目实战手册

本参考来自 `rr_teacher` 这类个人全栈项目的真实开发经验。使用时不要照搬技术栈，而要复用判断方式。

## 1. 用户闭环优先

先保护真实用户链路，再谈架构漂亮。`rr_teacher` 的主链路是老师拍题或上传图片，OCR 识别，人工复核，入库，组卷，导出 Word 并在微信中转发。

迁移到其他项目时，先写出一句话主链路：

```text
用户从哪里开始，经过哪些关键动作，最后得到什么可用结果？
```

只要主链路没有跑通，就不要把重点放在大规模重构、复杂后台或未来扩展上。

## 2. 先功能可用，再生产可控

个人项目常见节奏：

1. 跑通可用主链路。
2. 补配置治理、密钥、环境变量和部署说明。
3. 补成本控制、额度、日志和后台可见性。
4. 补数据库迁移、备份、回滚和生产演练。
5. 补服务层拆分、任务化和更稳定的长任务状态。

不要一开始就把所有问题都平台化；也不要在已经上线后继续只靠临时脚本和手动改库。

## 3. 不用重写逃避真实问题

`rr_teacher` 曾明确“不切 Java”：性能和成本问题不来自 Python 语言本身，而来自 OCR、图片处理、AI API、Word 生成、云托管费用、限流和任务超时。

做架构判断时先问：

- 当前痛点是不是语言或框架导致？
- 重写能否直接消除痛点？
- 迁移成本是否大于把现有链路补稳？
- 有没有更小的治理动作，例如限流、任务化、缓存、日志、部署参数？

## 4. 昂贵外部能力要产品化

OCR、AI、支付、云托管都不是“接上就完”。它们需要：

- 免费/低成本兜底。
- 高成本引擎额度。
- 用户级和全局级限制。
- 调用日志。
- 后台可见性。
- 失败提示和人工修正路径。

在 `rr_teacher` 中，Tesseract 适合低成本简单题，Pix2Text/豆包适合高精度或复杂题，因此需要额度、提示和后台用量明细。

## 5. AI/OCR 结果必须人工可复核

教学内容不能默认“AI 一次识别就正确”。识别结果页应方便查看原图、修改文本、快速保存；复杂题再进入完整编辑入库页。

对任何 AI 自动化功能都要检查：

- 用户能否看见原始输入和模型结果？
- 用户能否修正？
- 修正后保存的是最终可信内容，还是模型原始内容？
- 失败时是否有人工替代路径？

## 6. API 层要薄，业务下沉到 service

当 API 文件开始同时处理参数、权限、数据库关系、文件读写、外部服务和响应拼装时，就该下沉业务逻辑。

`rr_teacher` 已抽出：

- `QuestionService`：题目图片、标签读取与同步。
- `ExportService`：Word 导出附图读取和 payload 构建。
- 后续待补 `OcrService` 和任务状态模型。

判断标准：路由层应主要负责参数、权限和响应；核心业务逻辑应可单测。

## 7. 后台先服务维护者

个人项目的第一版后台不应默认做企业管理平台。优先解决维护者每天真实需要：

- 内容管理。
- 异常排查。
- 用户查看。
- 成本核对。
- 配额和套餐调整。
- 系统状态检查。

`rr_teacher` 选择个人开发者控制台，而不是一开始做学校、多租户、复杂 RBAC 或通用低代码。

## 8. 生产配置和迁移显式治理

上线前必须把危险默认值变成显式配置：

- 密钥不能使用开发占位值。
- 生产 CORS 和 Swagger 要受控。
- 生产 schema 变更要走迁移，不依赖 `create_all()`。
- `.env.example`、部署文档、CloudRun/CVM 配置模板要同步。
- 数据库迁移要有备份和回滚说明。

## 9. 验证工具失败时要换层级兜底

浏览器自动化、小程序自动化、云端环境都可能受工具链影响。不能因为一个工具跑不通就放弃验证，也不能把未验证写成已完成。

可替代验证层级：

- 单元测试。
- 静态检查或编译。
- API 级 smoke。
- 构建产物检查。
- 手工复核样例。
- 真实接口最小调用。

## 10. 用文档维护项目记忆

一个月以上的个人全栈项目需要可恢复上下文。重大功能、修复或阶段变化后更新：

- `progress.md`：当前状态、本轮完成、已知问题、下一步。
- `plan.md`：路线、架构、阶段计划变化。
- 专题文档：部署、迁移、后台、支付、Figma 等。

不要让下一轮开发只能靠聊天记录和记忆继续。
