# 交付检查清单

按任务风险选择需要的部分。不要机械全填；用它防漏项。

## 1. 需求与范围

- 是否写清目标用户和核心场景？
- 是否明确本轮必须交付的最小闭环？
- 是否列出非目标和延后项？
- 是否定义成功标准和验证方式？
- 是否有需要先问用户的关键歧义？

## 2. 架构与边界

- 是否先读取项目现有规则、目录结构、文档和测试？
- 是否复用现有技术栈、权限模型、部署方式和响应格式？
- 前端、后端、数据库、存储、第三方服务、部署目标是否都被识别？
- 是否避免引入未请求的大型抽象、微服务、多租户或复杂 RBAC？
- 是否说明新增模块和既有模块的边界？

## 3. 后端与 API

- 请求/响应模型是否定义在项目约定位置？
- 权限依赖是否正确使用？
- 是否返回项目统一响应格式？
- 路由是否只负责参数、权限、查询和响应？
- 复杂业务是否下沉到 service 或领域模块？
- 异步数据库流程中需要新 ID 时是否 `flush`？
- 是否避免循环懒加载和隐式 N+1？

## 4. 数据库与迁移

- 新字段、表、索引、默认值和约束是否清楚？
- dev SQLite 与 prod MySQL 的差异是否考虑？
- 是否有迁移脚本或 SQL，而不是依赖生产自动建表？
- 老数据是否保留？
- 失败回滚、备份点或人工恢复路径是否说明？
- 是否有针对迁移或模型的测试/检查？

## 5. 前端与用户体验

- 页面是否围绕真实用户流程，而不是演示空壳？
- 加载、空状态、错误状态、提交中、成功反馈是否覆盖？
- 表单校验是否和后端一致？
- 移动端/小程序/后台 Web 的交互入口是否符合各自场景？
- 管理后台是否先服务维护者，不提前做企业级复杂能力？

## 6. 外部服务、成本与支付

- 是否识别高成本或不稳定服务？
- 是否有额度、限流、日志和后台可见性？
- 是否有低成本或本地兜底？
- 支付场景是否以后端订单/回调/查询为准，而不是信任前端回调？
- 回调是否幂等？
- 密钥、商户号、API key 是否不写入代码和日志？

## 7. 部署与配置

- `.env.example` 或部署模板是否同步？
- 必填环境变量是否有启动校验？
- 生产密钥、CORS、Swagger、数据库连接是否受控？
- 构建产物是否进入正确静态目录或镜像？
- 健康检查、端口、启动命令是否验证？
- 是否有回滚建议或至少有上一版镜像/配置记录？

## 8. 验证矩阵

| 场景 | 推荐验证 |
| --- | --- |
| 新功能 | 单元测试 + API 测试 + 前端关键路径 |
| Bugfix | 最小复现 + 回归测试 |
| 数据库迁移 | 迁移命令 + 模型/接口读写 + 老数据检查 |
| 支付/成本 | 幂等测试 + 模拟回调 + 额度变化检查 |
| 后台功能 | HTTP smoke + 页面构建 + 关键交互 |
| 小程序功能 | 构建 + 页面逻辑测试 + 真机或开发者工具验证 |
| Word/文件导出 | 生成样例 + 文件内容或 media 包检查 |
| 部署上线 | health check + 关键 API smoke + 构建产物检查 |

## 9. 文档与交付

- 重大功能、重要修复或完整迭代是否更新 `progress.md`？
- 架构、路线或阶段计划变化是否更新 `plan.md`？
- 新增命令、环境变量、后台入口、部署步骤是否写入对应文档？
- 最终汇报是否区分已验证、未验证、已知风险？
- 是否避免把“代码已写”包装成“功能已完成”？

## 10. 可用性验收句式

交付前用下面句式自检：

```text
本轮完成后，用户可以在 [入口] 执行 [动作]，系统会通过 [后端/数据库/外部服务] 完成 [结果]。
我已用 [验证方式] 证明它可用。
仍未验证的是 [限制]，风险是 [风险]，下一步是 [动作]。
```
