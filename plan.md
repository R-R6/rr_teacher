# 高中化学教学辅助系统开发计划

## 1. 项目目标

面向高中化学老师，建设一套以“小程序快速录题 + 题库管理 + 试卷导出”为核心的教学辅助工具。

核心链路：

1. 老师拍题或上传图片；
2. 选择合适的 OCR 引擎完成识别；
3. 在识别结果页做快速复核；
4. 复杂题进入编辑入库页完整维护；
5. 题目沉淀到题库；
6. 通过手动/自动组卷导出 Word，并可在微信中转发。

## 2. 当前技术路线

### 前端

- `uni-app`
- Vue / JavaScript
- 微信小程序

### 后端

- FastAPI
- SQLAlchemy 2.0 async
- SQLite（开发）/ MySQL（生产）
- COS / 本地文件存储降级
- CloudBase CloudRun 容器托管

### OCR

- `tesseract`
  - 极速识别；
  - 适合纯文本、简单题、低成本录入。
- `pix2text_online`
  - 高精度识别（公式）；
  - 适合公式题、混排较轻的题目。
- `doubao_vision`
  - 高精度识别（复杂题）；
  - 适合流程图、装置图、复杂混排题。
- `pix2text_local`
  - 仅保留为本地开发/调试候选，不作为当前生产公开引擎。

### Word 导出

- `python-docx`
- 化学式 LaTeX/Unicode 转换
- 后端下载代理
- 小程序端 `wx.shareFileMessage` 转发 Word 文件

## 3. 架构原则

### 3.1 不切 Java

当前性能和成本问题不来自 Python 语言本身。项目强依赖 OCR、图片处理、AI API 和 Word 文档生成，Python 生态更合适。

Java 重写会带来 3-4 个月重建成本，且无法直接解决云托管费用、API 限流、任务超时等真实问题。

### 3.2 短期继续 CloudRun

CloudRun 已经跑通，手机端导出 Word、预览和转发也已验证。短期继续使用 CloudRun，重点补齐：

- 成本告警；
- 最大实例上限；
- 生产环境变量；
- 日志和监控；
- OCR/豆包调用额度。

### 3.3 CVM 作为中期备选

不立即迁移 CVM。先收集 1-2 周真实运行数据：

- 每日活跃老师数；
- OCR 次数；
- 豆包调用次数；
- Word 导出次数；
- CloudRun 账单；
- COS 存储与流量；
- 慢请求和错误率。

当 CloudRun 费用波动明显或运维策略需要固定成本时，再执行 CVM + Docker + Nginx 迁移。

### 3.4 从“功能可用”进入“生产可控”

下一阶段优先级不是继续堆功能，而是：

- 配置治理；
- 成本控制；
- 数据库迁移；
- 权限边界；
- 服务层拆分；
- 长任务任务化。
- 个人开发者控制台。

### 3.5 先做个人开发者控制台，不先做企业后台

当前项目已经具备教师端主链路，但缺少一个供个人开发者日常维护产品的正式入口。继续依赖小程序页面、临时接口和直接操作数据库，会让内容维护、异常排查、成本核对和用户查看越来越低效。

因此后台首期定位为“个人开发者控制台”，目标是让单个维护者可以在 Web 端完成日常管理，而不是一开始就做面向学校、运营或多租户的企业级后台。

首期约束：

- 复用现有 FastAPI 服务，不单独再建设一套后台后端；
- 新增独立 Web 管理台，不放在微信小程序端；
- 默认单人使用，不做复杂 RBAC、多租户、学校级数据隔离；
- 优先解决内容管理、运行监控、成本控制和异常处理。

## 4. 产品职责划分

### 识别结果页

职责：

- 查看原图；
- 查看识别结果；
- 修改少量错别字；
- 快速保存；
- 进入编辑入库。

不承担：

- 大段技术说明；
- 完整题目维护；
- 展示模型原始调试字段。

### 编辑入库页

职责：

- 完整编辑题目正文；
- 维护答案、解析、题型、难度、来源、标签；
- 维护题目附图；
- 补拍附图；
- 支持选择题中间缺失选项补位。

### 题目详情页

职责：

- 稳定展示题目、答案、解析、选项、附图；
- 支持附图预览；
- 为后续 Word 附图导出提供验证入口。

## 5. 题目附图策略

设计原则：

- 图不强行转成文字；
- 正文识别负责题干和选项；
- 流程图、装置图、结构图尽量作为附图保存；
- OCR 自动提取不稳定时，允许老师手动补拍。

链路：

1. OCR 结果返回 `images[]`；
2. 编辑入库页展示并允许删除/补拍；
3. 保存题目时挂接到 `QuestionImage`；
4. 详情页展示；
5. Word 导出时插入附图。

## 6. 里程碑规划

### Phase A：OCR 入库链路稳定（已完成第一版）

- 结果页正文规则统一；
- 多引擎识别结果回填；
- 正则解析选项；
- 保留原题内容中的选项文本；
- 支持中间缺失选项补位；
- 编辑页支持附图维护。

### Phase B：Word 导出转发（已完成第一版）

- 后端生成 Word；
- 后端下载代理；
- 小程序下载临时文件；
- 调用微信文件转发；
- 支持手机端转发到文件传输助手。

### Phase C：生产化配置治理（已完成第一版）

- 前端 API 地址环境化；
- 后端密钥环境化；
- 整理 `.env.example`、`.env.docker`、`.env.cloud.example`；
- CloudRun 环境变量模板补齐；
- 部署文档补齐。

### Phase D：成本控制与限流（已完成第一版）

- 用户级 OCR 日额度；
- 全局豆包预算；
- 超额提示切换到 Tesseract；
- 数据库限流记录；
- CloudRun 最大实例和费用告警记录仍需运维侧继续核对。

### Phase E：数据库迁移治理（下一步 P0）

- 引入 Alembic；
- 创建当前 schema baseline；
- 生产不再依赖自动 `create_all()` 修改表结构；
- 建立备份与回滚流程。

### Phase F：Word 附图导出（已完成第一版）

- 导出题目附图；
- 修正图片读取方式，避免依赖临时签名 URL；
- 统一题目图片顺序；
- 验证试卷卷、答案卷、直接题目导出。

### Phase G：服务层拆分与任务化（进行中 P1）

- 已抽出 `QuestionService`，承接题目图片/标签读取与同步逻辑；
- 已抽出 `ExportService`，承接 Word 导出附图读取与 payload 构建逻辑；
- 待补 `OcrService`；
- 待定义任务状态模型；
- OCR/导出后续逐步改为异步任务。

### Phase H：个人开发者控制台（已完成第一版，待浏览器验收）

- 已落地后台管理 API、独立 `admin-web` 工程、首期页面与 `/admin-console` 静态挂载；
- 已提供后台白名单控制、HTTP 级 smoke 脚本与基础测试入口；
- 已补齐用户详情中的 OCR 明细、付费引擎额度状态与用户套餐/每日/月度 OCR 限额后台配置底座；
- 剩余工作聚焦浏览器自动化烟雾验证、交互细节收口与必要说明文档补充；
- 暂不引入多租户、学校管理、复杂角色权限和通用低代码能力。

## 7. 下一步执行计划

### 第 1 步：微信登录与真实微信支付联调

目标：打通 `openid` 获取、真实商户配置和低金额真实支付验收，让种子计划 billing 从“底座已接好”进入“可实测”状态。

任务：

- 完成小程序微信登录，并确认后端用户可稳定拿到 `openid`；
- 配置 `WECHAT_APPID`、`WECHAT_SECRET` 与 `WECHAT_PAY_*` 商户参数、平台证书、公网 HTTPS `notify URL`；
- 用低金额真实订单验证 JSAPI 下单、支付回调验签、订单状态更新、权益发放和后台 billing 可见性；
- 记录 `pending`、缺 `openid` 返回 400、回调验签失败等排查路径。

### 第 2 步：个人开发者控制台浏览器验收与收口

目标：在已完成第一版交付和用户套餐/限额底座的基础上，补齐浏览器层验证，让后台可作为个人开发者日常控制台稳定使用。

任务：

- 在可用浏览器自动化环境下补一轮登录、仪表盘、题库、标签、OCR 详情烟雾测试；
- 补一轮用户详情中的套餐限额配置、OCR 明细与额度状态回归；
- 根据浏览器烟雾结果修补首批交互细节；
- 视需要补充后台使用说明或部署说明。

### 第 3 步：服务层拆分与任务化

目标：降低 API 层复杂度，为 OCR、导出等长任务打基础。

任务：

- 补齐 `OcrService`；
- 在已落地 `QuestionService` / `ExportService` 基础上继续下沉 OCR、题目管理和导出路由逻辑；
- 设计任务状态模型；
- OCR/导出逐步改为异步任务。

### 第 4 步：数据库迁移生产演练

目标：验证生产 schema 变更流程可备份、可回滚。

任务：

- 记录备份流程；
- 在测试环境执行 Alembic upgrade/current/history；
- 记录失败回滚步骤。

### 第 5 步：CloudRun 成本与实例策略核对

目标：用真实运行数据决定是否继续 CloudRun 或迁移 CVM。

任务：

- 核对 CloudRun 最大实例和费用告警配置；
- 收集 OCR、导出、COS 的真实调用量和费用；
- 形成是否迁移 CVM 的判断记录。
