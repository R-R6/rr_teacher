# 高中化学教学辅助系统开发计划

## 1. 项目目标

面向高中化学老师，建设一套以“小程序快速录题 + 题库管理 + 试卷导出”为核心的教学辅助工具。

核心链路：

1. 老师拍题或上传图片；
2. 选择合适的 OCR 引擎完成识别；
3. 在识别结果页做快速复核；
4. 复杂题进入编辑入库页完整维护；
5. 题目沉淀到题库；
6. 通过手动/自动组卷导出 Word，并可在微信中转发。

## 2. 当前技术路线

### 前端

- `uni-app`
- Vue / JavaScript
- 微信小程序

### 后端

- FastAPI
- SQLAlchemy 2.0 async
- SQLite（开发）/ MySQL（生产）
- COS / 本地文件存储降级
- CloudBase CloudRun 容器托管

### OCR

- `tesseract`
  - 极速识别；
  - 适合纯文本、简单题、低成本录入。
- `pix2text_online`
  - 高精度识别（公式）；
  - 适合公式题、混排较轻的题目。
- `doubao_vision`
  - 高精度识别（复杂题）；
  - 适合流程图、装置图、复杂混排题。
- `pix2text_local`
  - 仅保留为本地开发/调试候选，不作为当前生产公开引擎。

### Word 导出

- `python-docx`
- 化学式 LaTeX/Unicode 转换
- 后端下载代理
- 小程序端 `wx.shareFileMessage` 转发 Word 文件

## 3. 架构原则

### 3.1 不切 Java

当前性能和成本问题不来自 Python 语言本身。项目强依赖 OCR、图片处理、AI API 和 Word 文档生成，Python 生态更合适。

Java 重写会带来 3-4 个月重建成本，且无法直接解决云托管费用、API 限流、任务超时等真实问题。

### 3.2 短期继续 CloudRun

CloudRun 已经跑通，手机端导出 Word、预览和转发也已验证。短期继续使用 CloudRun，重点补齐：

- 成本告警；
- 最大实例上限；
- 生产环境变量；
- 日志和监控；
- OCR/豆包调用额度。

### 3.3 CVM 作为中期备选

不立即迁移 CVM。先收集 1-2 周真实运行数据：

- 每日活跃老师数；
- OCR 次数；
- 豆包调用次数；
- Word 导出次数；
- CloudRun 账单；
- COS 存储与流量；
- 慢请求和错误率。

当 CloudRun 费用波动明显或运维策略需要固定成本时，再执行 CVM + Docker + Nginx 迁移。

### 3.4 从“功能可用”进入“生产可控”

下一阶段优先级不是继续堆功能，而是：

- 配置治理；
- 成本控制；
- 数据库迁移；
- 权限边界；
- 服务层拆分；
- 长任务任务化。

## 4. 产品职责划分

### 识别结果页

职责：

- 查看原图；
- 查看识别结果；
- 修改少量错别字；
- 快速保存；
- 进入编辑入库。

不承担：

- 大段技术说明；
- 完整题目维护；
- 展示模型原始调试字段。

### 编辑入库页

职责：

- 完整编辑题目正文；
- 维护答案、解析、题型、难度、来源、标签；
- 维护题目附图；
- 补拍附图；
- 支持选择题中间缺失选项补位。

### 题目详情页

职责：

- 稳定展示题目、答案、解析、选项、附图；
- 支持附图预览；
- 为后续 Word 附图导出提供验证入口。

## 5. 题目附图策略

设计原则：

- 图不强行转成文字；
- 正文识别负责题干和选项；
- 流程图、装置图、结构图尽量作为附图保存；
- OCR 自动提取不稳定时，允许老师手动补拍。

链路：

1. OCR 结果返回 `images[]`；
2. 编辑入库页展示并允许删除/补拍；
3. 保存题目时挂接到 `QuestionImage`；
4. 详情页展示；
5. Word 导出时插入附图。

## 6. 里程碑规划

### Phase A：OCR 入库链路稳定（已完成第一版）

- 结果页正文规则统一；
- 多引擎识别结果回填；
- 正则解析选项；
- 保留原题内容中的选项文本；
- 支持中间缺失选项补位；
- 编辑页支持附图维护。

### Phase B：Word 导出转发（已完成第一版）

- 后端生成 Word；
- 后端下载代理；
- 小程序下载临时文件；
- 调用微信文件转发；
- 支持手机端转发到文件传输助手。

### Phase C：生产化配置治理（下一步 P0）

- 前端 API 地址环境化；
- 后端密钥环境化；
- 整理 `.env.example`、`.env.docker`、`.env.cloud.example`；
- CloudRun 环境变量核对；
- 部署文档补齐。

### Phase D：成本控制与限流（下一步 P0）

- 用户级 OCR 日额度；
- 全局豆包预算；
- 超额降级到 Tesseract；
- Redis 或数据库限流；
- CloudRun 最大实例和费用告警记录。

### Phase E：数据库迁移治理（下一步 P0）

- 引入 Alembic；
- 创建当前 schema baseline；
- 生产不再依赖自动 `create_all()` 修改表结构；
- 建立备份与回滚流程。

### Phase F：Word 附图导出（P1）

- 导出题目附图；
- 修正图片读取方式，避免依赖临时签名 URL；
- 统一题目图片顺序；
- 验证试卷卷、答案卷、直接题目导出。

### Phase G：服务层拆分与任务化（P1）

- 抽 `OcrService`；
- 抽 `QuestionService`；
- 抽 `ExportService`；
- 定义任务状态模型；
- OCR/导出逐步改为异步任务。

## 7. 下一步执行计划

### 第 1 步：配置治理

目标：让本地、CloudRun、未来 CVM 可以使用不同配置，不需要手改源码。

任务：

- 改造前端 API_BASE；
- 检查后端默认密钥；
- 补齐生产环境变量说明；
- 更新部署文档。

### 第 2 步：OCR 成本控制

目标：防止豆包调用失控。

任务：

- 设计 `ocr_quota`；
- 限制单用户每日豆包次数；
- 增加全局每日调用上限；
- 前端展示额度耗尽提示。

### 第 3 步：数据库迁移

目标：让生产 schema 可控。

任务：

- 初始化 Alembic；
- 生成 baseline；
- 写明迁移命令；
- 关闭生产隐式建表策略。

### 第 4 步：Word 附图导出

目标：让含图题目真正进入 Word。

任务：

- 后端读取 `QuestionImage`；
- 插入题目附图；
- 测试手机端预览和转发；
- 补导出测试。

## 8. 当前最优先事项

1. 完成配置治理。
2. 完成 OCR/豆包调用额度。
3. 引入数据库迁移。
4. 继续验证 Word 导出和题目附图导出。
5. 收集真实运行成本，再决定是否迁移 CVM。
